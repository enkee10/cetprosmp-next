## New collection model

Collection: `comentarios`

Document ID: `{comentarioId}`

Fields:

- `id`: string, required, immutable, must equal document ID.
- `postId`: string, required, immutable, Data Connect post ID stored as string.
- `parentId`: string or null, required, immutable. Root comments use null; replies reference a root comment.
- `nivel`: number, required, immutable. Allowed values: 0 root, 1 reply.
- `texto`: string, required, editable by author, max 2000 chars.
- `autorUid`: string, required, immutable, must equal `request.auth.uid`.
- `autorNombre`: string, required, immutable, max 80 chars.
- `autorFoto`: string URL or null, required, immutable.
- `estado`: string, required. Allowed: `visible`, `pendiente`, `oculto`, `eliminado`.
- `creadoEn`: timestamp, required, immutable, must be recent on create.
- `actualizadoEn`: timestamp, required, must be recent on create/update.
- `moderadoPorUid`: string or null, required. Users create with null; moderators set their own UID.

## Intended access

- Read/list comments: authenticated users only.
- Create comments: authenticated users only.
- Create root comment: `nivel == 0` and `parentId == null`.
- Create reply: `nivel == 1`, `parentId` points to an existing root comment for the same `postId`.
- Author update: only `texto` and `actualizadoEn`.
- Moderator update: users with `level >= 300` can update `estado`, `moderadoPorUid`, `texto`, and `actualizadoEn`.
- Delete: users with `level >= 300` only.

## Devil's advocate checks

- Public list exploit: blocked by `isAuthenticated()`.
- Unauthorized read/write: reads require auth; writes require owner on create/update or moderator.
- Update bypass: full validator runs on create and update.
- Ownership hijacking: `autorUid` must equal auth UID on create and is immutable on update.
- Immutable fields: `id`, `postId`, `parentId`, `nivel`, author identity, and `creadoEn` are immutable.
- Type juggling: validator checks strings, numbers, booleans via null/string/timestamp constraints.
- Resource exhaustion: `texto`, `autorNombre`, `autorFoto`, `postId`, `parentId`, `estado`, `id`, `autorUid`, `moderadoPorUid` have size limits.
- Schema pollution: `hasOnlyAllowedFields` blocks extra fields.
- Timestamp manipulation: create/update timestamps must be recent and not in the future.
- Thread depth bypass: replies can only target an existing comment with `nivel == 0`.
- Cross-post reply spoofing: parent comment `postId` must match request `postId`.
- Comment activation bypass: Firestore rules cannot read PostgreSQL/Data Connect `posts.comentarios_activos`; app/server code must check it before creating comments.
